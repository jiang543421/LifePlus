# LifePulse v2.1.0-ai — Release Notes

> MVP2 第五阶段：**AI 分析 v2.1 LLM 增强** —— 在 v2.0 的 Provider 聚合模板底座之上叠加 LLM 调用 + 3 层降级链路 + 独立分析页 /ai-analysis + 配额 / 熔断 + 启动期 fail-fast。
> **Tag**: `v2.1.0-ai`（annotated）on `main` @ `4d81aa5`（squash merge of `feat/ai-llm-v2.1`）。
> **Date**: 2026-07-22。
> **对比基线**: `v2.0.0-ai`（`946995b` PR #16 squash merge）。

---

## 1. Highlights

- **LLM 真接入**：v2.0 是 5 Provider 聚合 + 模板引擎；v2.1 在不破坏既有架构的前提下新增 `LlmInsightGenerator`（L1），结果经 `LlmJsonParser` 字段校验 + 敏感词扫描；失败无重试（5xx/4xx/超时统一走 L2 模板）。
- **3 层降级链路**（hard rule，CLAUDE.md §11.3）：
  ```
  L1 LLM 成功 → source="llm"，渲染完整 payload
  L1 LLM 失败 → catch Llm*Exception → L2 模板 → source="template"
  L2 模板失败 → throw BusinessException(1501) → HTTP 503
  ```
  用户感知最小化：source 标签 + 缺段占位，**无错误弹窗**。
- **6h Redis 缓存 + 3/min 限流**：cache TTL 从 30min 升到 6h（CLAUDE.md §11.4 — 与首页 + 独立页共享 key）；`POST /refresh` rate limit 从 v2.0 的 6/min 收紧到 **3/min/user**（防 LLM 重放刷量）。
- **配额 + 熔断**：默认 50 LLM 调用/用户/天；熔断 10 失败/5min 触发，30min 恢复。Redis 不可用：quota **fail-open**（放行）/ circuit **fail-closed**（不熔断）/ cache **MISS**（走计算），分别匹配业务语义。
- **双 LLM Provider**：`provider=deepseek`（OpenAI 兼容协议，`POST /chat/completions`）/ `provider=ollama`（原生 Ollama chat API，本地进程）。Ollama 模式自动 bypass 熔断（本地进程死掉与远程故障语义不同）。
- **独立分析页 `/ai-analysis`**：v2.0 仅有抽屉的一屏入口；v2.1 新增专属路由，由首页抽屉底部「查看完整分析 →」跳转进入。渲染 4 段（headline / advice / highlight / chips）+ mood + ElAlert 降级提示 + ElEmpty 空态。
- **`source` 角标（LLM 智能 / 模板）**：HomeView AI 卡右上角绝对定位角标，让用户在不打开抽屉的情况下就感知"本次洞察是 LLM 真生成还是模板降级"。
- **5 个新错误码**（spec 11.6）：1510 配额超限 / 1511 熔断中 / 1512 解析失败 / 1513 不可用；外部表现几乎不可见（Service 全 catch 走 L2），仅日志 / metrics 暴露。
- **零破坏性 schema**：CLAUDE.md §11.1 硬约束 1-2：「无新表 + 不修改既有字段」— 5 张业务表（t_task / t_plan / t_expense / t_diet / t_daily_report）字段语义不变。
- **零新第三方依赖**：CLAUDE.md §11.1 硬约束 3 — LLM 客户端用手写 `RestClient` 调 OpenAI 兼容协议，不引入 Spring AI 库。pom.xml 干净。
- **TDD 闭环**：T1-T15 全部 Red→Green→Refactor；5 个 hotfix 累积（`@Component` 移除 / `withFreshness` 9-arg 保住 v2.1 字段 / `LlmQuotaGuard` ctor 标注 / `AbstractIntegrationTest` 默认值 / `AI_DEGRADED` HTTP 503）；无 TODO/FIXME 残留。
- **gitleaks pre-commit（PR4）**：`.githooks/pre-commit` + `.gitleaks.toml` 新增 3 条规则（DeepSeek API key / LP_JWT_SECRET / 高熵 base64）；开发者克隆后跑 `./scripts/install-hooks.sh` 设置 `core.hooksPath`，提交期拦截。

| 编排 | `LlmInsightGenerator.generate(userId)` — quota check → circuit check → call client → parse → return；任一环抛 `Llm*Exception` 由调用方 catch |
| Service 改造 | `AiInsightService` — 注入 `LlmInsightGenerator`；`getInsight(userId)` 先 L1，catch 走 L2 模板，注入 `source` 字段；TTL 30min → 6h；`withFreshness` 9-arg 版本保留所有 v2.1 字段（hotfix #2）|
| Controller | `AiInsightController` — 新增 `GET /api/v1/ai/insight/analysis`（与 `/today` 共享缓存 + 30/min/user 限流，复用 `InsightGetRateLimit`）；refresh 限流 6/min → 3/min |
| 异常码 | `ErrorCode` 追加 `LLM_QUOTA_EXCEEDED=1510` / `LLM_CIRCUIT_OPEN=1511` / `LLM_INVALID_RESPONSE=1512` / `LLM_UNAVAILABLE=1513`；原 `AI_DEGRADED=1501` HTTP 状态 500 → 503（hotfix #4）|
| 安全 | 全部按 `UserContext.current()` 过滤（spec §11.1 硬约束 6：`{id}` 不接受 userId 形参，由 JWT 解析得）|
| 配置注入 | `application.yml` 增加 `lp.ai.llm.*` 配置块，全部走 `${LP_LLM_*:}` 占位符；`/docker-compose.yml` 新增 `ollama` 服务（开发态）与 `backend` 服务的 `LP_LLM_*` env 注入 |
| 密钥管理 | fail-fast 在 `LlmProperties.@PostConstruct`：校验 4 项（enabled / provider / apiKey presence / apiKey length）；**Ollama 模式**自动放宽（key 可空，但 host 必须配置）|

## 5. 升级指南（v2.0.0-ai → v2.1.0-ai）

### 5.1 兼容性

- ✅ **无破坏性 API 变更**（既有 4 个 AI 端点 100% 兼容，前端 v2.0 调用照旧可用）
- ✅ **无 schema 变更**（CLAUDE.md §11.1 硬约束；AI 模块纯增量，只读 5 张既有表）
- ✅ **0 个新 Flyway 迁移**
- ✅ **无前端路由冲突**（`/api/v1/ai/insight/analysis` 与既有 `/today` 并列）
- ✅ **无 CI 闸门硬性变更**（沿用 `mvn verify` + `pnpm vitest` + `pnpm exec playwright test`）
- ⚠️ **新增环境变量**：`LP_LLM_PROVIDER`、`LP_LLM_API_KEY`、`LP_LLM_BASE_URL`、`LP_LLM_MODEL`、`LP_LLM_DAILY_QUOTA`、`LP_LLM_CIRCUIT_BREAKER_*` — 详见 `.env.example`
- ⚠️ **启动期更严**：`provider=deepseek` 时缺失/占位 `LP_LLM_API_KEY` 或长度 < 20 → Spring 启动失败；务必先填 `.env` 再 `docker compose up -d --build`

### 5.2 升级步骤

```bash
# 1. 拉取 tag
git fetch origin
git checkout v2.1.0-ai

# 2. 配置 LLM（首次启用必须；Ollama 模式可跳过 API key）
cp .env.example .env
# 编辑 .env：填 LP_LLM_API_KEY=sk-xxx（DeepSeek 控制台创建）
# 或选择 Ollama：LP_LLM_PROVIDER=ollama + LP_LLM_BASE_URL=http://ollama:11434

# 3. 启动 Ollama 服务（仅 Ollama 模式需要；DeepSeek 模式跳过）
docker compose up -d ollama
docker compose exec ollama ollama pull llama3.1:8b   # 首次下载 ~4.7GB

# 4. 启动主栈
docker compose up -d --build

# 5. 启用 pre-commit（推荐 — 拦截密钥误提交）
./scripts/install-hooks.sh    # Windows 需 Git Bash 或 WSL

# 6. 验证后端健康
curl -fsS http://localhost:8080/actuator/health
# 期望：{"status":"UP",...} 且 logs 无 "LP_LLM_API_KEY must be set"

# 7. 验证 AI 端点（v2.0 老路径仍可用）
curl -fsS -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  http://localhost:8080/api/v1/ai/insight/today | jq .
# 期望：code=0；data.source ∈ ["llm","template"]；data.headline 非空

# 8. 验证 v2.1 新端点
curl -fsS -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  http://localhost:8080/api/v1/ai/insight/analysis | jq .
# 期望：code=0；data.source / advice / highlight / mood 至少 source 有值
#   source=llm   → advice/highlight/mood 都非空
#   source=template → advice/highlight 为空，mood 为空；1501 路径则整包空
```

### 5.3 回滚步骤

```bash
# 1. 停止服务
docker compose down

# 2. 回退代码
git checkout v2.0.0-ai
docker compose up -d --build

# 3. 清理 v2.1 缓存（可选，无害，不影响 v2.0）
docker compose exec redis redis-cli \
  --scan --pattern 'lp:ai:*' | xargs -r redis-cli DEL
```

### 5.4 数据迁移提示

- **无 schema 变更**；旧数据无需迁移
- 升级前**无需**手工操作
- 升级后**新增**的能力：
  1. AI 洞察可由 LLM 真生成（DeepSeek / Ollama）
  2. 用户感知：「AI 智能」vs「模板」角标 + 完整 4 段独立分析页
  3. 配额熔断防御：50/day/user 上限 + 10 失败/5min 熔断
  4. 6h 缓存复用（首页抽屉 / 独立分析页同缓存）
- **首次激活体验**：本地无 DeepSeek key → 自动走 Ollama 模式或纯模板（取决于 env）；用户首次打开抽屉可能看到的是模板结果（source=template），这是预期的渐进式 UX，不是 bug

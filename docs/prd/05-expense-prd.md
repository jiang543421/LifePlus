## 1. 产品目标（SMART）

| 维度 | 内容 |
|---|---|
| **Specific（具体）** | 在 LifePulse 内提供「单笔消费录入 + 月度汇总 + 分类占比」完整闭环，替代首页占位卡，使消费记录不再依赖外部记账 App |
| **Measurable（量化）** | 单笔录入端到端 ≤ 1 秒（P95）；列表接口 P95 ≤ 200ms；月汇总接口 P95 ≤ 300ms；ECharts 分类环图渲染 ≤ 500ms |
| **Achievable（可达）** | 沿用 MVP1 Spring Boot 3 + MyBatis-Plus + Element Plus 既有架构；不引入新基础设施；单开发者 1 个 sprint 可完成 |
| **Relevant（相关）** | 与 04-future §2.2「日常开支记账与统计」一致；为 Phase 2.4 AI 分析模块积累 ≥30 天消费数据 |
| **Time-bound（时限）** | v1.2.1 release；目标与 settings v1.1 同一 release train；后续 v1.2.2 = 饮食模块串行发布 |

## 2. 用户画像

基于 [00-index.md](./00-index.md) §4 三类核心用户画像，本模块聚焦其中两类，并新增一类「轻量记账」专有场景。

### 画像 A：「忙碌的独立工作者·小李」

- **特征**：30 岁独立产品经理，节奏自主，月度餐饮 + 交通 + 订阅支出频繁但金额不大；希望「随手记一笔」而非精细财务。
- **痛点**：在多平台切来切去（微信账单 / Apple Wallet / Notion 模板），月底想看「这个月到底餐饮花了多少」时数据散落各处，无单一视图。
- **核心任务**：① 出门消费后 30 秒内记一笔；② 月底打开 LifePulse 一眼看到分类占比；③ 误录一笔后能快速修改或删除。

### 画像 B：「远程办公·小张」

- **特征**：28 岁远程研发工程师，订阅服务多（云服务、域名、Copilot、Notion、ChatGPT、VPN 等）；希望看清「钱都流到哪里」。
- **痛点**：现有记账 App 对「订阅」分类要么缺失要么不直观，无法一眼识别哪些订阅可以砍。
- **核心任务**：① 标记一笔订阅为 SUBSCRIPTION；② 按 SUBSCRIPTION 筛选，看所有订阅清单；③ 月底看 SUBSCRIPTION 占比是否过大。

### 画像 C：「记账新人·小林」（新增）

- **特征**：25 岁应届毕业生，第一次主动记账；不愿下载 / 学习 / 付费专业记账 App。
- **痛点**：专业 App 注册繁琐、强制手机号、学习曲线高；想要「能用就行」。
- **核心任务**：① 注册即用（沿用 LifePulse 已有邮箱注册）；② 不必选预算 / 账户 / 币种这些专业概念；③ 5 笔 / 月也愿意坚持记录。

## 3. 功能清单（MVP + 扩展）

### 3.1 MVP（v1.2.1 必做）

**录入 / 编辑**

- [ ] 创建一笔消费（金额必填 > 0、分类必选 5 选 1、备注可选 ≤200 字符、发生时间必填）
- [ ] 编辑已有消费（金额 / 分类 / 备注 / 发生时间任意字段可改；空字段不更新）
- [ ] 软删除消费（30 天内可恢复 — *注：MVP1 暂不实现「恢复面板」，仅软删，下次启动同 MVP1 task 模块做法*）
- [ ] 单笔消费录入端到端 ≤ 1 秒（含提交 + 列表更新 + Toast 反馈）

**列表 / 筛选**

- [ ] 查看消费列表（按发生时间倒序，按天分组）
- [ ] 按分类筛选（5 个预置分类可单选 / 全选）
- [ ] 按时间区间筛选（from / to，最长跨度 1 年）
- [ ] 列表分页（每页 20 条；总条数上限 100/页）

**月汇总**

- [ ] 查看当月总支出
- [ ] 查看当月分类占比（ECharts 环图，5 个分类百分比 + 绝对值）
- [ ] 查看同比上月（百分比 + 绝对值差额）
- [ ] 查看同比去年同月（百分比 + 绝对值差额）
- [ ] 切换月份（向前 / 向后翻月，URL 同步）

**首页集成**

- [ ] 首页「消费」卡 placeholder 切换为 false（v1.2.1 发布时合并）
- [ ] 首页「消费」卡点击 → 跳 `/expenses`

**安全 / 隔离**

- [ ] 所有 `{id}` 端点强制按 `user_id` 过滤（跨用户越权返回 1003）
- [ ] 写端点限流 10 次 / 分钟 / userId（Redis Lua + fail-closed）

### 3.2 扩展（v1.2.1 不做）

- [ ] 月度 / 分类预算（超预算提醒）
- [ ] 循环账单自动生成（订阅按月）
- [ ] 多账户（信用卡 / 借记 / 现金）
- [ ] 转账（账户间）
- [ ] 多币种 + 汇率换算
- [ ] 自定义分类（用户增删预置之外的分类）
- [ ] 导入 / 导出（CSV、微信账单、支付宝账单）
- [ ] 跨年热力图 + 季度 / 年度趋势分析
- [ ] AI 趋势洞察（依赖 Phase 2.4，需 ≥30 天数据沉淀，PRD 见 04-future §2.4）

### 3.3 逻辑说明：KANO / RICE 如何影响划分

**KANO 模型映射**（参考 [00-index.md §5.1](./00-index.md)）：

| 层级 | 含义 | 本模块对应 |
|---|---|---|
| 基本（Must-be） | 缺失即模块不可用 | CRUD、跨用户隔离、软删、统一错误码 |
| 期望（One-dimensional） | 越多越好 | 月汇总、分类占比、筛选、时间分组 |
| 兴奋（Attractive） | 缺失不致命 | 同比、环比、ECharts 环图视觉 |
| 无差异（Indifferent） | 用户不在意 | 同色环图配色、图表尺寸 |

→ **MVP 包含「基本 + 期望」**；「兴奋」中同比环比因实施成本低（≈4 行 SQL）一并纳入；视觉细节进 v1.3+ polish。

**RICE 排序**（参考 [00-index.md §5.2](./00-index.md) 通用框架）：

| 候选 | Reach | Impact | Confidence | Effort | 决策 |
|---|---|---|---|---|---|
| 单笔 CRUD | 100% | 高 | 高 | 低 | ✅ MVP |
| 列表 + 筛选 | 100% | 高 | 高 | 中 | ✅ MVP |
| 月汇总 + 分类占比 | 80% | 高 | 高 | 中 | ✅ MVP |
| 同比 / 环比 | 60% | 中 | 高 | 低 | ✅ MVP（成本低） |
| 预算 | 40% | 中 | 中 | 中 | ⏸ v1.3 |
| 循环账单 | 30% | 中 | 中 | 中 | ⏸ v1.3 |
| 自定义分类 | 20% | 低 | 中 | 中 | ⏸ v2.x |
| 导入 / 导出 | 30% | 低 | 中 | 高 | ⏸ v2.x |

→ RICE 排序后预算 / 循环 / 自定义分类 / 导入导出均落 v1.3+ 或 v2.x，与 PRD 04-future §4 启动节奏一致。

## 4. 用户故事

### 故事 EXP-1：30 秒内随手录入一笔消费

> 作为 **忙碌的独立工作者·小李**，我需要 **在外出消费后 30 秒内完成录入**，以便于 **不打断当前工作流、降低记账摩擦**。

**验收标准（AC）**：
1. 录入表单仅含 4 个字段（金额 / 分类 / 备注 / 发生时间）；必填项实时校验、错误提示 < 200ms 内弹出。
2. 默认发生时间为「当前时间 - 5 分钟」（照顾出门场景）；可手动改。
3. 提交成功后立即关弹窗 + 列表顶部新增该项 + Toast「已记录 ¥XX」；端到端 ≤ 1 秒。

### 故事 EXP-2：月底一眼看清当月花了多少 / 各类占比

> 作为 **「忙碌的独立工作者·小李」/「远程办公·小张」**，我需要 **在月底打开 LifePulse 一眼看到当月总支出 + 各分类占比 + 同比**，以便于 **判断这个月钱花得是否合理、是否需要调整**。

**验收标准（AC）**：
1. 列表页右侧固定显示「当月总支出」大数字 + ECharts 分类环图（5 个分类百分比 + 绝对值），无数据时显示空态文案。
2. 同比上月 / 同比去年同月各显示一行（如「+12%」「-5%」），无历史数据时显示「无对比数据」而非破折号。
3. 可点击月份向前 / 向后翻页，URL 同步 `?year=YYYY&month=MM`；刷新页面保留月份上下文。

### 故事 EXP-3：按订阅分类查看所有订阅清单

> 作为 **远程办公·小张**，我需要 **按 SUBSCRIPTION 分类筛选查看所有订阅**，以便于 **识别哪些订阅可砍、合计占月支出多少**。

**验收标准（AC）**：
1. 列表筛选条分类下拉含「全部 / 餐饮 / 购物 / 交通 / 订阅 / 其他」6 个选项，单选生效。
2. 选中「订阅」后，列表仅显示 `category=SUBSCRIPTION` 的消费项；月度总支出数字相应刷新。
3. 切换分类后 URL 同步 `?category=SUBSCRIPTION`；可分享 / 收藏该 URL。

### 故事 EXP-4：误录一笔后快速修改或删除

> 作为 **所有用户**，我需要 **能够修改或删除已录入的消费**，以便于 **纠正误录（金额 / 分类 / 时间错）**。

**验收标准（AC）**：
1. 列表每行右侧有「编辑」「删除」按钮；编辑打开 dialog 复用新增表单（mode='edit'），回显当前值。
2. 删除走 ElMessageBox 二次确认（输入框要求输入「确认注销」风格文案可简化为「确定删除」）；确认后调 DELETE，成功移除该行 + Toast。
3. 编辑 / 删除越权时（虽然列表不会显示他人数据，但 URL 直访问他人 `{id}`）返回 1003，UI 跳回列表 + Toast「资源不存在」。

## 5. 功能范围与边界

### 5.1 In Scope（v1.2.1 必做）

| 项 | 详情 |
|---|---|
| 数据模型 | `t_expense` 表（id / user_id / amount / category / note / occurred_at / 审计字段 / @TableLogic） |
| API | 7 个端点（GET 列表 / GET 单个 / POST / PATCH / DELETE / GET summary / GET categories） |
| 前端 | ExpenseView 两栏布局 + ExpenseDialog 录入 + ExpenseDetailView 编辑 / 删除 |
| 图表 | ECharts 分类占比环图（仅 ExpenseView 使用，路由级 code-split 避免污染首页） |
| 安全 | 跨用户 1003、限流 10/min/userId、SQL 注入防护、错误日志脱敏 |
| 测试 | Service 单测 ≥15 cases、WebMvc slice ≥10 cases、IT ≥6 cases、Vitest store ≥6 / 组件 ≥8 / Dialog ≥4、E2E ≥3 cases |

### 5.2 Out of Scope（明确不做）

| 项 | 不做的原因 |
|---|---|
| 预算 / 超预算提醒 | RICE Effort=中、Impact=中、Confidence=中，且会引入「预算重置」「周期概念」等模型复杂度 → v1.3 |
| 循环账单（订阅） | 与「订阅分类」语义重叠；用户手动记一笔已足够低频 → v1.3 |
| 多账户 / 转账 | 个人工具 + 个人账本，账户模型复杂（卡组织、币种、对账） → v2.x |
| 自定义分类 | 预置 5 个覆盖 MVP 场景；后期评估用户实际诉求 → v2.x |
| 导入 / 导出 | 涉及多格式解析（CSV / Excel / 微信账单 PDF）→ v2.x |
| AI 趋势洞察 | 依赖 ≥30 天数据沉淀（04-future §4 Phase 2.4）→ 不在本模块 |
| 图片识别 / OCR | CLAUDE.md §6.2 显式不做 |
| 多语言 | MVP1 仅中文；分类中文文案硬编码，枚举英文名对齐后端 |

## 6. 成功指标（KPIs）

### 6.1 性能指标

| 指标 | 目标 |
|---|---|
| 单笔录入端到端 P95（含 UI 反馈） | ≤ 1.0 秒 |
| 列表接口 P95（GET /api/v1/expenses） | ≤ 200 毫秒 |
| 月汇总接口 P95（GET /api/v1/expenses/summary） | ≤ 300 毫秒 |
| ECharts 分类环图渲染（路由级 code-split 后） | ≤ 500 毫秒 |
| 首页加载 P95（消费 v1.2.1 发布后） | ≤ 2.0 秒（不因新增首页卡片退化） |

### 6.2 安全指标

| 指标 | 目标 |
|---|---|
| 跨用户越权拦截率 | 100%（单元 / 切片 / IT 全覆盖） |
| 写端点限流触发率（误伤） | < 1% 正常用户 |
| 错误日志脱敏覆盖率 | 100%（不打 note 全文、不打 password / token） |

### 6.3 质量指标

| 指标 | 目标 |
|---|---|
| 后端 ExpenseService 行覆盖 | ≥ 80%（JaCoCo 闸门） |
| ExpenseController 切片覆盖 | 鉴权 / 校验 / 越权路径 100% |
| 前端 store 行覆盖 | ≥ 70% |
| 关键组件（ExpenseView / ExpenseDialog）覆盖 | ≥ 80% |
| E2E 关键流 | ≥ 3 cases 通过 |

### 6.4 业务指标（上线后 4 周观察）

| 指标 | 目标 |
|---|---|
| 新功能启用率（v1.2.1 发布后 7 天内有过至少 1 笔消费记录的用户 / 活跃用户） | ≥ 30% |
| 月活人均月记录笔数 | ≥ 10 笔（验证「30 秒录入」门槛合理） |
| 90 天留存（连续 90 天有记录的用户 / 首月记录用户） | ≥ 20%（个人工具基线） |

## 7. 风险与应对

| 类别 | 风险 | 等级 | 应对 |
|---|---|---|---|
| 技术 | ECharts 包体（~280KB gzip）影响首页 P95 超 2.0s | 中 | 首页不直接 `import echarts`；ExpenseView 内 `await import('echarts/core')` 动态加载；路由级 code-split |
| 技术 | `DECIMAL(12,2)` 在前端 JS Number 失精 | 低 | 前端展示用字符串 / `decimal.js`；服务端始终 `BigDecimal` 计算；金额接口序列化为字符串而非 number |
| 技术 | 月汇总 SQL 性能（聚合 5 分类 + 同比） | 低 | `idx_user_category (user_id, category, occurred_at)` 覆盖；同月查询 LIMIT 5 行；同比查询用 DATE_SUB 函数索引扫描 |
| 资源 | 单笔录入摩擦高于预期，用户不愿用 | 高 | 上线后 4 周观察「月人均记录笔数」；若 < 5 笔 / 月，触发 UX 复审（缩短录入字段 / 加语音 / 加短信触发等） |
| 资源 | 限流阈值（10/min）误伤高频用户 | 低 | 阈值 10/min 较 settings（10/min 写）略宽；前端 1006 错误文案「操作过于频繁，请稍后再试」友好提示；观察误伤率 < 1% |
| 市场 | 已有记账 App（鲨鱼记账 / Money Pro 等）功能深度远超 | 中 | 定位差异：LifePulse 是「个人数字生活」统一入口而非专业记账；强调任务 / 日程 / 消费 / 饮食在同仓单点；不追求记账深度 |
| 演进 | 分类后期想扩展（如「医疗」「教育」） | 低 | 迁移路径已明确：`ExpenseCategory` enum → `t_dict_category` 字典表；本期不预埋，避免 schema 过早固化 |

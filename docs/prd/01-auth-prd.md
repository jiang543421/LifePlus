## 1. 模块概述

认证模块是 LifePulse 的入口与安全基座，承担三件事：

1. **让新用户进入**：邮箱 + 密码注册即用，零等待。
2. **让老用户回来**：登录后 7 天内不需要重新输入密码。
3. **把数据分清**：每位用户只能看到自己的任务与日程，跨用户访问一律拒绝。

整个模块是「基本需求」级别的存在——任何缺失都会让产品无法使用。

## 2. 功能清单（MVP）

### 2.1 注册与登录

- [ ] 新用户邮箱注册（含昵称可选）
- [ ] 邮箱 + 密码登录
- [ ] 登录失败次数过多时拒绝（同 IP/邮箱 1 分钟内 ≥6 次）
- [ ] 登录态自动续期（无感刷新会话）
- [ ] 主动退出登录（撤销当前会话）

### 2.2 账号管理

- [ ] 查看当前账号信息（邮箱、昵称、注册时间）
- [ ] 修改昵称（在设置页）
- [ ] 修改密码（在设置页，校验旧密码）
- [ ] 账号注销（软删除账号与所有数据，30 天后清理）

### 2.3 安全基线

- [ ] 密码以不可逆强哈希方式存储（永远不以明文落库）
- [ ] 会话凭证哈希后存储，登出/重放/换设备可即时失效
- [ ] 跨用户访问拦截（任意资源接口带身份时按账号过滤，越权拒绝）
- [ ] 统一错误码（前端不依赖 HTTP 状态码判断业务失败）
- [ ] 登录限流（同 IP/邮箱 1 分钟内 ≥6 次必拒）

## 3. 用户故事

### 故事 AUTH-1：新用户注册
> 作为 **未注册用户**，我需要 **用邮箱与密码注册账号**，以便于 **开始使用 LifePulse 记录自己的任务与日程**。

**验收标准（AC）**：
1. 提交合法邮箱 + 8 位以上（含字母与数字）的密码后，注册成功并自动登录到首页。
2. 邮箱已注册时，注册页明确提示「该邮箱已注册」，不暴露是否可登录。
3. 密码不足 8 位 / 仅字母 / 仅数字，注册页明确指出具体不满足的规则。

### 故事 AUTH-2：老用户登录
> 作为 **已注册用户**，我需要 **用邮箱与密码登录**，以便于 **继续访问我之前的任务与日程数据**。

**验收标准（AC）**：
1. 输入正确邮箱 + 密码，登录成功，跳转到上次访问的页面或首页。
2. 同 IP/邮箱在 1 分钟内连续失败 6 次，第 6 次返回明确「请求过于频繁，请稍后再试」。
3. 输入错误密码，提示信息不区分「邮箱不存在」与「密码错误」，防止账号枚举。

### 故事 AUTH-3：会话保持与退出
> 作为 **已登录用户**，我需要 **系统在后台自动续期我的登录态，但退出后立即失效**，以便于 **不用频繁重新登录，但能随时保护隐私**。

**验收标准（AC）**：
1. 用户连续 7 天活跃，期间不需要重新输入密码。
2. 用户点击「退出登录」后，立刻无法访问任何受保护页面。
3. 退出时仅撤销当前会话，不影响其他已登录设备。

## 4. KPI

| 指标 | 目标 |
|---|---|
| 注册流程端到端成功率（含表单校验） | ≥ 99% |
| 登录流程端到端成功率 | ≥ 99% |
| 跨用户越权拦截率 | 100% |
| 登录限流命中正确率 | 100% |
| 续期凭证重放拒绝率 | 100% |
| 会话自动续期失败导致用户被迫重新登录的次数 | 0 / 7 天 |

## 5. 边界

**In Scope**：邮箱注册 / 登录 / 自动续期 / 退出 / 当前账号 / 修改昵称 / 修改密码 / 账号注销 / 登录限流 / 跨用户拦截。

**Out of Scope**：手机号注册 / 第三方社交登录 / 双因素认证 / 找回密码（邮箱验证码）/ 单点登录。**找回密码**优先级次于 Phase 1 全部模块，待 Phase 1.4 后评估。

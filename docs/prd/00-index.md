## 1. 产品概述

**LifePulse = 个人多用户「数字生活」仪表盘**。把分散在多个工具里的日常管理——任务、日程——收拢到一个轻量 Web 应用内，让个人用户的「今天要做什么 / 这周有哪些安排」一眼可见；未来可扩展到日报、消费、饮食、AI 分析。

**MVP1 目标**：交付一个可独立运行、可注册登录、可记录任务、可管理日程的最小可用产品，并为 Phase 2 模块预留占位入口。

## 4. 用户画像

### 画像 A：忙碌的独立工作者「小李」
- **身份**：30 岁独立产品经理，自由职业，节奏自主但日程碎片化。
- **痛点**：每天在 4 个工具间切换（记事本、邮箱、日历 App、微信收藏），经常漏掉承诺事项；任务和日程不在一个视图里。
- **核心任务**：记录当日承诺、规划本周客户会议、随时回看节奏。
- **典型场景**：周日晚上列下周待办 + 标优先级；周一上午在首页一眼看到今天的 3 件事和 2 个会议。

### 画像 B：自我驱动型学生「小王」
- **身份**：22 岁计算机专业大四学生，准备考研与期末。
- **痛点**：学习任务多（网课、刷题、论文），临时课程安排总被打乱。
- **核心任务**：按截止日查看任务、按周/月查看课程与自习、把任务绑定到自习时段。
- **典型场景**：选课周录每周固定课程；为「线性代数复习」建任务并绑定到某次自习。

### 画像 C：远程办公的「小张」
- **身份**：28 岁远程研发工程师，时区弹性但会议密集。
- **痛点**：工作与个人事件混在公司日历里太杂；想要一个「只属于自己的」轻量记录工具。
- **核心任务**：记录私人任务（健身、社保续缴）、安排私人日程（约会、医疗预约）。
- **典型场景**：开完会后把私人待办写进 LifePulse；周末浏览本月日历规划家庭聚餐。

## 5. 决策逻辑

### 5.1 KANO 模型与 MVP / 扩展划分

| 需求层级 | 含义 | MVP 归属 |
|---|---|---|
| 基本需求（Must-be） | 缺失即产品不可用：账号体系、数据隔离、会话安全 | **MVP 必做** |
| 期望需求（One-dimensional） | 越多越好：任务/日程的核心 CRUD、筛选、日历视图 | **MVP 必做** |
| 兴奋需求（Attractive） | 有了惊喜但缺失不致命：AI 趋势分析、智能提醒 | **Phase 2** |
| 无差异需求（Indifferent） | 用户不在意：界面主题色、动画细节 | 当前不做 |

### 5.2 RICE 模型与功能优先级

| 候选 | Reach | Impact | Confidence | Effort | 排序 |
|---|---|---|---|---|---|
| 认证（注册/登录/会话） | 100% | 高 | 高 | 中 | 🥇 第一梯队 |
| 任务 CRUD + 筛选 | 100% | 高 | 高 | 中 | 🥇 第一梯队 |
| 日历事件 CRUD + 月视图 | 100% | 高 | 高 | 中 | 🥇 第一梯队 |
| 首页卡片与跳转 | 100% | 中 | 高 | 低 | 🥈 第二梯队 |
| 任务 ↔ 日程关联 | 60% | 中 | 高 | 低 | 🥈 第二梯队 |
| AI 分析 | 100% | 理论高 | 低 | 高 | 🥉 Phase 2 |
| 日报自动生成 | 80% | 中 | 中 | 高 | 🥉 Phase 2 |
| 推送通知 | 70% | 中 | 高 | 中 | 🥉 Phase 2 |

**关键决策示例**：AI 分析虽然评分高，但 Confidence 低——AI 建议依赖任务/日程/消费数据沉淀，MVP1 没有这些数据，强行上线只会给出空泛结论，故放 Phase 2。

## 6. 范围与边界

### 6.1 In Scope（MVP1 必须交付）

- **账号体系**：注册、登录、自动续期、退出、当前账号、修改昵称、修改密码、账号注销
- **任务模块**：CRUD、状态切换、软删除、按状态/优先级/标签/截止日筛选、分页、详情、按所属日程查看、首页计数
- **日程模块**：CRUD、软删除、月历视图、按月范围查询、分页、详情、新建/编辑弹窗、全天事件、提醒时长占位字段
- **首页**：6 张卡片网格 + 占位卡提示、顶栏、响应式断点、未登录拦截
- **安全**：密码强哈希、登录限流、跨用户访问拒绝、统一错误码、统一返回信封

### 6.2 Out of Scope（本期明确不做）

| 模块 | 不做的原因 |
|---|---|
| 团队/共享/权限分级 | 个人工具，引入共享会带来权限模型复杂度，超出 MVP 范围 |
| 图片上传与视觉识别 | 未设计附件表；OCR/识别需第三方服务 |
| 离线缓存/PWA | 增加前端状态复杂度，需权衡投入产出比 |
| 支付与订阅 | 个人工具，盈利模型不在 MVP 范围 |
| 多语言 | MVP1 仅中文，避免维护成本 |
| 日报 / 消费 / 饮食 / AI 分析 | Phase 2；首页仅占位 |
| 推送通知 | 提醒时长仅记录字段，不发推送 |
| 移动端原生 App | MVP1 仅 Web 响应式 |
| 数据导入/导出 | Phase 2 视用户反馈决定 |

### 6.3 边界判定原则

- **不引入新模块**——任何新增模块归到 Phase 2 储备。
- **不修改既有数据模型的字段语义**——Phase 2 扩展通过新迁移脚本加默认值，不动既有数据。
- **不破坏基本需求安全基线**——任何变更必须保留：密码哈希、会话哈希、限流、跨用户拦截、统一错误码。

## 8. 风险与应对（高优先级 8 条）

| 类别 | 风险 | 等级 | 应对 |

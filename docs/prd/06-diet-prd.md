## 1. 产品目标（SMART）

| 维度 | 内容 |
|---|---|
| **Specific（具体）** | 在 LifePulse 内提供「单笔饮食记录 + 当日营养汇总 + 一键复用」完整闭环，替代首页占位卡，使用户在 LifePulse 内即可追踪每日摄入而不必依赖专业食物库 App |
| **Measurable（量化）** | 单笔录入端到端 ≤ 1 秒（P95）；列表接口 P95 ≤ 200ms；当日汇总接口 P95 ≤ 250ms；ECharts 营养柱图渲染 ≤ 500ms；「一键复用」点击到填值 ≤ 100ms（纯前端） |
| **Achievable（可达）** | 沿用 MVP1 Spring Boot 3 + MyBatis-Plus + Element Plus 既有架构；ECharts 与消费模块共享代码切片；单开发者 1 个 sprint 可完成；v1.2.1（消费）上线后再启动 |
| **Relevant（相关）** | 与 04-future §2.3「每日饮食记录 + 营养摄入估算」一致；为 Phase 2.4 AI 分析模块积累 ≥30 天饮食数据；与消费模块完全独立（不互通），避免模块耦合导致任一延期阻塞另一个 |
| **Time-bound（时限）** | v1.2.2 release；与 v1.2.1（消费）串行发布；在 v1.2.1 合并后启动，期间可收集 v1.2.1 用户反馈以微调 v1.2.2 设计 |

## 2. 用户画像

基于 [00-index.md](./00-index.md) §4 三类核心用户画像，本模块复用其中 1 类并新增 2 类专有场景（健身 / 慢病管理）。

### 画像 A：「健身爱好者·小陈」（新增）

- **特征**：25 岁办公室 + 健身房 3-4 次/周，目前增肌期（target 蛋白质 ≥ 120g/日）；对外卖 + 健身餐切换频繁，对每餐的「蛋白质 / 碳水」具体克数有量化诉求。
- **痛点**：薄荷健康 / 食物库 App 数据全但录入繁琐；想要「蛋白质 35g」快速入账而非逐项查食物数据库。
- **核心任务**：① 餐后 30 秒内记一笔（蛋白质 / 热量手动填、名称自由文本）；② 一眼看清「今天蛋白质还差多少」；③ 常用食物（燕麦 / 鸡胸肉 / 米饭）一键复用。

### 画像 B：「慢病饮食管理·老王」（新增）

- **特征**：55 岁退休，三高（高血压 / 高血脂 / 糖尿病前期），医生要求「控盐 < 5g/日、控糖主食减半」；独居，自己做饭为主。
- **痛点**：专业营养 App 学习曲线陡、术语多（宏量营养素 / 升糖指数）；想要「简单几项数字」足够即可。
- **核心任务**：① 主食 / 加餐随手记录；② 周末看「这周糖类摄入是否过多」；③ 不被强制食物数据库绑定。

### 画像 C：「独立工作者·小李」（沿用 00-index §4 画像 A，聚焦饮食场景）

- **特征**：30 岁独立产品经理，节奏快，外卖为主；想看清「自己每天到底吃了什么、有没有重油重盐」，但不愿精细到克。
- **痛点**：饮食无规律，时常「中午忙没吃 / 晚上暴饮暴食」；想要个轻量记录让自己「看见」。
- **核心任务**：① 一日三餐随手记；② 当日营养卡显示「vs 昨日」差值，发现突变；③ 一周后回看「这一周餐饮结构」。

## 3. 功能清单（MVP + 扩展）

### 3.1 MVP（v1.2.2 必做）

**录入 / 编辑**

- [ ] 创建一笔饮食（餐别 4 选 1、名称必填 ≤64 字符、4 项营养可填 0、备注可选 ≤200 字符、发生时间必填）
- [ ] 编辑已有饮食（餐别 / 名称 / 营养 / 备注 / 时间任意字段可改；空字段不更新）
- [ ] 软删除饮食（沿用 MVP1 task / plan 软删做法）
- [ ] 单笔录入端到端 ≤ 1 秒（含提交 + 列表更新 + Toast 反馈）
- [ ] 「一键复用」按钮：弹下拉显示近 30 天 top 10 高频名称，点击自动填入名称 + 平均营养；端到端 ≤ 100ms（纯前端缓存命中）

**列表 / 筛选**

- [ ] 按天分组列表（按 `occurred_at` 日期分组，每组按餐别 + 时间排序）
- [ ] 按餐别筛选（早 / 午 / 晚 / 加餐 单选）
- [ ] 按日期筛选（默认今日；可前后切日，URL 同步 `?date=YYYY-MM-DD`）
- [ ] 列表分页（每页 20 条；总条数上限 100/页）

**当日营养汇总**

- [ ] 查看当日 4 项营养合计（kcal / protein_g / carb_g / fat_g）
- [ ] 横向柱图对比推荐摄入常量（kcal=2000 / protein=60 / carb=300 / fat=65）
- [ ] vs 昨日差值（如 +120 kcal / -5g 蛋白）
- [ ] vs 上周同日差值（如 -80 kcal）
- [ ] 无对比数据时显示「无对比数据」文案而非破折号

**首页集成**

- [ ] 首页「饮食」卡 placeholder 切换为 false（v1.2.2 发布时合并）
- [ ] 首页「饮食」卡点击 → 跳 `/diets`

**安全 / 隔离**

- [ ] 所有 `{id}` 端点强制按 `user_id` 过滤（跨用户越权返回 1003）
- [ ] 写端点限流 10 次 / 分钟 / userId（与消费模块阈值对齐）

**模块独立**

- [ ] 与消费模块完全独立：不共享外键、不共享 API、不共享前端组件（数据互通留待 v2.x，04-future §2.3 决策）

### 3.2 扩展（v1.2.2 不做）

- [ ] 内置食物库（~5000 条中国食物营养表，自动按份量计算）
- [ ] 拍照识别 / OCR（CLAUDE.md §6.2 显式不做）
- [ ] 个性化推荐摄入常量（按性别 / 体重 / 年龄 / 活动量）
- [ ] 智能饮食建议（依赖 Phase 2.4 AI 分析模块，需 ≥30 天数据沉淀）
- [ ] 周 / 月营养趋势图（折线 / 堆叠）
- [ ] 食物过敏 / 忌口标签
- [ ] 自定义餐别（如「下午茶」「宵夜」独立分类）
- [ ] 与消费模块互通（一笔餐饮消费同步记入饮食）
- [ ] 营养目标设定（用户自定 kcal 目标，超 / 欠提示）
- [ ] 营养师 / 医生协作面板

### 3.3 逻辑说明：KANO / RICE 如何影响划分

**KANO 模型映射**（参考 [00-index.md §5.1](./00-index.md)）：

| 层级 | 含义 | 本模块对应 |
|---|---|---|
| 基本（Must-be） | 缺失即模块不可用 | CRUD、跨用户隔离、软删、统一错误码 |
| 期望（One-dimensional） | 越多越好 | 当日汇总、按天分组、餐别筛选 |
| 兴奋（Attractive） | 缺失不致命 | 「一键复用」、vs 昨日 / 上周同日对比、推荐摄入柱图 |
| 无差异（Indifferent） | 用户不在意 | 推荐常量数值是否精确（人群均值足够） |

→ **MVP 包含「基本 + 期望」**；「一键复用」虽属 Attractive 但实施成本极低（≈1 个 SQL 聚合 + 1 个组件）故纳入 MVP；周 / 月趋势（兴奋度高但 Effort 中）延后到 v1.3。

**RICE 排序**（参考 [00-index.md §5.2](./00-index.md) 通用框架）：

| 候选 | Reach | Impact | Confidence | Effort | 决策 |
|---|---|---|---|---|---|
| 单笔 CRUD | 100% | 高 | 高 | 低 | ✅ MVP |
| 当日 4 项营养汇总 | 90% | 高 | 高 | 中 | ✅ MVP |
| 「一键复用」 | 70% | 中 | 中 | 低 | ✅ MVP（低成本） |
| vs 昨日 / 上周同日 | 60% | 中 | 高 | 低 | ✅ MVP（低成本） |
| 按餐别筛选 | 80% | 中 | 高 | 低 | ✅ MVP |
| 周 / 月趋势 | 40% | 中 | 中 | 中 | ⏸ v1.3 |
| 食物库（内置） | 50% | 高 | 中 | 高 | ⏸ v2.x（数据源 + 维护成本） |
| OCR / 拍照 | 30% | 中 | 中 | 高 | ⏸ v2.x 或 never（CLAUDE §6.2 显式不做） |
| 个性化推荐常量 | 30% | 中 | 中 | 中 | ⏸ v2.x（需 user_profile 表） |

→ 食物库与 OCR 均因 Effort=高 + 与 YAGNI 原则不符落 v2.x+；其中 OCR 已在 CLAUDE.md §6.2 显式不做，记录在 backlog。

## 4. 用户故事

### 故事 DIET-1：30 秒内记录午餐

> 作为 **健身爱好者·小陈**，我需要 **在餐后 30 秒内完成午餐记录（含 4 项营养数字）**，以便于 **不过度打断训练 / 工作流、保持记录连贯性**。

**验收标准（AC）**：
1. 录入表单含 7 个字段（餐别 / 名称 / 4 项营养 / 备注 / 时间）；发生时间默认「现在 - 5 分钟」；营养字段支持 0（用于「吃了但忘了热量」场景）。
2. 「一键复用」下拉默认展示近 30 天 top 10 高频名称（含平均营养），点击自动填入；纯前端缓存命中时 ≤ 100ms。
3. 提交成功后立即关弹窗 + 列表顶部新增该项 + 当日营养卡 4 项数字同步刷新；端到端 ≤ 1 秒。

### 故事 DIET-2：当日营养卡一眼看出还差多少

> 作为 **健身爱好者·小陈** / **独立工作者·小李**，我需要 **在打开饮食页时一眼看到今日 4 项营养对比推荐常量的差距**，以便于 **快速判断「今天蛋白质还差多少」「热量是否超标」**。

**验收标准（AC）**：
1. 列表页右侧固定显示「当日营养卡」：4 项数字（实际 / 推荐）+ 横向柱图比例可视化。
2. 柱图超 100% 用红色 / 未达 60% 用橙色 / 60-100% 用绿色三档色阶提示（无障碍：不依赖颜色，附 ▲▼ 文本指示）。
3. vs 昨日 / vs 上周同日各一行（如「+120 kcal」「-5g 蛋白」），无历史数据时显示「无对比数据」而非破折号。

### 故事 DIET-3：常用食物一键复用

> 作为 **独立工作者·小李** / **健身爱好者·小陈**，我需要 **对常吃的食物（早餐燕麦 / 午餐鸡胸肉）一键复用上次记录**，以便于 **减少重复输入、提高连续记录意愿**。

**验收标准（AC）**：
1. 录入弹窗顶部有「一键复用」下拉；数据源 `GET /api/v1/diets/frequent?from=&to=&limit=10`（默认近 30 天）。
2. 点击下拉某项后，名称 + 4 项平均营养自动填入表单；用户可在此基础上微调（修改某项后仍提交为新记录）。
3. 复用的「平均营养」基于该名称所有近 30 天历史记录均值；无历史时该名称不出现。

### 故事 DIET-4：误录一笔后快速修改或删除

> 作为 **所有用户**，我需要 **能够修改或删除已录入的饮食**，以便于 **纠正误录（餐别错 / 数字估错）**。

**验收标准（AC）**：
1. 列表每行右侧有「编辑」「删除」按钮；编辑打开 dialog 复用新增表单（mode='edit'），回显当前值。
2. 删除走 ElMessageBox 二次确认；确认后调 DELETE，成功移除该行 + 当日营养卡 4 项数字同步减少 + Toast。
3. 编辑 / 删除越权时（URL 直访问他人 `{id}`）返回 1003，UI 跳回列表 + Toast「资源不存在」。

## 5. 功能范围与边界

### 5.1 In Scope（v1.2.2 必做）

| 项 | 详情 |
|---|---|
| 数据模型 | `t_diet` 表（id / user_id / meal_type / name / 4 项营养 / note / occurred_at / 审计字段 / @TableLogic） |
| API | 7 个端点（GET 列表 / GET 单个 / POST / PATCH / DELETE / GET summary / GET frequent） |
| 前端 | DietView 两栏布局 + DietDialog 录入（含「一键复用」下拉）+ DietDetailView 编辑 / 删除 + DietDayGroup / DietNutritionCard 子组件 |
| 图表 | ECharts 营养横向柱图（仅 DietView 使用，与消费共享路由级 code-split 切片） |
| 推荐常量 | `REC_DAILY_KCAL=2000` / `REC_DAILY_PROTEIN_G=60` / `REC_DAILY_CARB_G=300` / `REC_DAILY_FAT_G=65`（人群均值，硬编码于 `constants/diet.ts`） |
| 安全 | 跨用户 1003、限流 10/min/userId、SQL 注入防护、错误日志脱敏 |
| 测试 | Service 单测 ≥15 cases、WebMvc slice ≥10 cases、IT ≥6 cases、Vitest store ≥6 / 组件 ≥8 / Dialog ≥4、E2E ≥3 cases |

### 5.2 Out of Scope（明确不做）

| 项 | 不做的原因 |
|---|---|
| 内置食物库 | 数据源 + 维护成本高；MVP2 用户量未到需要此深度的阶段；先验证「自由文本 + 手填」是否够用 → v2.x |
| 拍照 / OCR 识别 | CLAUDE.md §6.2 显式不做 |
| 个性化推荐常量 | 需 user_profile 表 + 字段；MVP2 不引入新表 → v2.x |
| 周 / 月营养趋势图 | 数据要求 ≥7 天连续记录，MVP2 初期数据稀疏，趋势失真 → v1.3 |
| 智能饮食建议 | 依赖 Phase 2.4 AI 分析模块（需 ≥30 天数据沉淀） → 不在本模块 |
| 食物过敏 / 忌口标签 | 涉及过敏原数据 + 警告规则 → v2.x |
| 自定义餐别 | 预置 4 个覆盖 MVP 场景；SNACK 餐别可重复使用 → v2.x |
| 与消费模块互通 | 04-future §2.3 决策为「完全独立」；互通留待 v2.x 评估「同一天记录跳转」方案 |
| 多语言 | MVP1 仅中文；餐别 + 文案中文硬编码，枚举英文名对齐后端 |

## 6. 成功指标（KPIs）

### 6.1 性能指标

| 指标 | 目标 |
|---|---|
| 单笔录入端到端 P95（含 UI 反馈） | ≤ 1.0 秒 |

| 指标 | 目标 |
|---|---|
| 跨用户越权拦截率 | 100% |
| 写端点限流触发率（误伤） | < 1% 正常用户 |
| 错误日志脱敏覆盖率 | 100%（不打 note 全文、不打 name 全文） |

| 指标 | 目标 |
|---|---|
| 后端 DietService 行覆盖 | ≥ 80%（JaCoCo 闸门） |

| 前端 store 行覆盖 | ≥ 70% |
| 关键组件（DietView / DietDialog / DietNutritionCard）覆盖 | ≥ 80% |
| E2E 关键流 | ≥ 3 cases 通过 |

### 6.4 业务指标（上线后 4 周观察）

| 指标 | 目标 |
|---|---|
| 新功能启用率（v1.2.2 发布后 7 天内有过至少 1 笔饮食记录的用户 / 活跃用户） | ≥ 25%（略低于消费：饮食需更高频记录门槛） |

| 「一键复用」使用率（点击复用下拉的提交次数 / 总提交次数） | ≥ 40%（验证复用价值） |
| 90 天留存（连续 90 天有记录的用户 / 首月记录用户） | ≥ 15%（饮食门槛高于消费，预期略低） |

## 7. 风险与应对

| 类别 | 风险 | 等级 | 应对 |
|---|---|---|---|
| 技术 | ECharts 包体（与消费共享 280KB 切片）影响首屏 P95 | 中 | 与消费共用同一路由级 code-split chunk；DietView 同样按需 `import()`；不引入第二种图表库 |
| 技术 | Frequent 接口 N+1（聚合全表） | 低 | `idx_user_occurred (user_id, occurred_at)` 覆盖；默认 30 天窗口 + LIMIT 10；IT 验证 10 万级数据下 P95 < 250ms |
| 技术 | 当日 4 项营养聚合 SQL 性能 | 低 | 现有索引覆盖；单日窗口过滤；同表内聚合（无需 JOIN） |
| 资源 | 营养手填用户嫌麻烦 → 数据稀疏 | 高 | 上线后 4 周观察「周人均记录笔数」；若 < 7 笔/周，触发 UX 复审（一键复用强化 / 营养估算辅助 / 简化表单等） |
| 资源 | 推荐常量（2000/60/300/65）非个性化，健身 / 慢病人群不匹配 | 中 | MVP2 用人群均值；柱图比例对极端用户不直观；后期 v2.x 加 user_profile 字段后个性化 |
| 资源 | 餐别 4 选与用户习惯冲突（下午茶 / 宵夜） | 低 | SNACK 餐别可重复使用（早 + 下午茶均归 SNACK）；不在本模块扩展 |
| 市场 | 薄荷 / 食物库 App 数据深度远超 | 中 | 定位差异：LifePulse 是「个人数字生活」统一入口而非专业营养；强调「轻量 + 一键复用 + 跨模块」价值 |
| 演进 | 食物库后期想集成 | 低 | 数据源（如薄荷 API / 食物库 CSV）作为 v2.x 独立模块；t_diet 表 `name` 字段预留，未来可加 `food_id` 外键 |
| 演进 | 与消费模块互通 | 低 | 决策已落「v2.x 评估同一天跳转方案」；本期不预留 FK，避免早期 schema 固化 |

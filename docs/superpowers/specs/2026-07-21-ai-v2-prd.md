## 1. 产品目标（SMART）

### 1.1 一句话价值主张

**让用户打开首页的第一眼，就能 5 秒内判断"今天的状态如何"**——无需打开 4 个子页面。

### 1.2 SMART 目标

| 维度 | 内容 |
|---|---|
| **S** Specific | 把首页"AI 分析"占位卡升级为智能洞察卡，基于 5 类数据（任务/计划/消费/饮食/日报）自动生成 1-2 句中文洞察 + 3 个关键指标 chip，并提供详情抽屉 |
| **M** Measurable | 智能卡在登录后 0.5s 内显示 headline（缓存命中路径）；Service 行覆盖 ≥ 85%；E2E 4 个核心路径全绿；无 401/越权用例失败 |
| **A** Achievable | 复用日报 `MetricProvider` 模式 + 5 个数据源已存在（task/plan/expense/diet/daily），纯规则+模板无 LLM 风险，1 人 2-3 周可完成 |
| **R** Relevant | 直接服务"个人多用户数字生活仪表盘"定位；与 MVP1（任务/计划）+ v1.2.1（消费）+ v1.2.3（日报）形成"数据采集 → 智能汇总"完整链路 |
| **T** Time-bound | 2026-08-11 前完成 v2.0 全量开发；2026-08-15 前完成 E2E + 文档；2026-08-18 提 PR review |

## 2. 用户画像

> 个人应用，目标用户为项目所有者（开发者本人）。为辅助设计与测试，构造 3 个典型使用场景。

### 2.1 场景 A：连续记录 30 天的"高活跃用户"

- **背景**：每天在 LifePulse 记录 5-10 个任务、2-4 个计划、3-5 笔消费、2-3 餐饮食；偶尔生成日报
- **痛点**：
  - 想快速判断"今天能收尾吗 / 这周是否超额"但需要打开 4 个子页面
  - 每天登录第 1 分钟在 4 个页面间切换，决策疲劳
- **核心任务**：
  - 早晨 8 点打开首页 → 看 AI 卡 → 决定"今天主攻任务 X"
  - 周日晚 9 点看本周消费趋势
- **成功标志**：AI 卡显示准确洞察（如"任务堆积 3 个"），无需打开子页面即知状况

### 2.2 场景 B：刚注册 3 天的"新用户"

- **背景**：注册后录入了 2-3 个任务，尚未生成消费/饮食数据
- **痛点**：
  - AI 卡显示"？"或"无数据"会让用户怀疑"这产品没价值"
- **核心任务**：
  - 看 AI 卡 → 看到"还没有数据，继续记录几天后将出现洞察" → 知道"要继续录数据"
- **成功标志**：空状态文案清晰传达"录入越多洞察越准"的预期

### 2.3 场景 C："日报模块未上线时的混合场景"

- **背景**：v2.0 上线时，日报 v1.2.3 尚未合并到 main
- **痛点**：
  - 如果 AI 模块硬依赖日报，会导致 v2.0 整体不可用
  - 如果抛 503，用户体验差
- **核心任务**：
  - 用户看到 AI 卡正常显示任务/计划/消费/饮食 4 维洞察，**不知道也无感**日报未上线
- **成功标志**：AI 卡 headline + 3 chip 全部正常返回，1501 错误码从未触发

### 3.4 KANO × RICE 划分逻辑

#### KANO 模型映射

| KANO 类别 | 对应需求 | 处置 |
|---|---|---|
| **基本型 (Must-have)** | 智能卡渲染 / 鉴权 / 越权 / 5 数据源接入 / 错误码 1006 | 全部进 MVP；缺失则产品无核心价值 |
| **期望型 (Expected)** | 30 min 缓存 / 手动刷新 / 详情抽屉 / 空状态文案 | 高 RICE 进 MVP；中等 RICE 进 v2.1 |
| **魅力型 (Excitement)** | 趋势图 / LLM / 反馈 / 历史 | 全部进 v2.2+ |

#### RICE 评分（v2.0 子任务抽样）

> 个人项目 R=1。RICE = (1 × I × C) / E。I ∈ {0.25, 0.5, 1, 2, 3}，C ∈ {50%-100%}，E 单位"人天"。

| 任务 | I（影响）| C（信心）| E（人天）| RICE | 处置 |
|---|---|---|---|---|---|
| T5.1 Service 编排 | 3（核心）| 80% | 1.0 | 2.4 | MVP |
| T4.2 Task Provider | 3（基础数据）| 90% | 0.4 | 6.75 | MVP |
| T4.6 Daily Provider（降级）| 2（防 503）| 90% | 0.4 | 4.5 | MVP（防御性必须）|
| T9.5 AiInsightCard | 3（首屏）| 80% | 0.6 | 4.0 | MVP |
| T10.1 E2E | 2（回归保障）| 90% | 0.4 | 4.5 | MVP |
| E1 独立分析页 | 3 | 50% | 5.0 | 0.30 | 扩展（RICE 低）|
| E2 趋势迷你图 | 1 | 60% | 2.0 | 0.30 | 扩展 |
| E14 LLM 改写 | 2 | 30% | 8.0 | 0.075 | 扩展（C 极低，外部依赖）|

**为什么 T4.6 必入 MVP？** RICE 看似与其他 Provider 持平，但它是**"v2.0 整体可用性"的保险丝**——如果不做降级，v1.2.3 未合并时 AI 卡 503 概率高，会拉低整个产品的可靠性叙事。属"防御性必做"。

**为什么 E1（独立分析页）放扩展？** RICE 0.30 主要拖累在 E（5 人天）和 C（50%——多 Tab 设计未验证）。若先在 v2.0 通过抽屉收用户反馈（v2.1 调研），RICE 可重新评估。

**为什么 E14（LLM）放最末？** C=30%（外部 API 稳定性 + 成本未知），引入密钥/限流/降级架构，E=8 人天，与"1 人 + 显式不做"原则冲突，**应在 v2.0 跑通纯规则数据后再决定**。

## 4. 用户故事列表

> "作为 [用户]，我需要 [能力]，以便于 [价值]" + 3 条验收标准（AC）。

### 4.1 US-1：每日状态总览

**作为** 每天打开 LifePulse 的用户，**我需要** 在首页看到 AI 自动生成的当日状态总结（1-2 句中文），**以便于** 不必打开 4 个子页面也能判断"今天能不能收尾 / 钱花得是否理性"。

**AC**：
- AC-1.1 登录后访问 `/`，在 1.5s 内（缓存命中 0.5s / 未命中 1.0s）看到 AI 卡显示 headline 文本（非占位）
- AC-1.2 headline 字数 40-120 字，1-2 句中文，标点正确
- AC-1.3 headline 内容至少引用 1 个有效数据源（如"任务完成率 X%"），不为空字符串

### 4.2 US-2：3 个关键指标 chip

**作为** 用户，**我需要** 在 AI 卡上看到 3 个横排的关键指标（任务完成率 / 本周消费 / 今日日程），**以便于** 快速扫描当日核心数字。

**AC**：
- AC-2.1 卡面固定显示 3 个 chip，顺序固定为 `taskCompletion → weeklyExpense → planDensity`
- AC-2.2 每个 chip 包含 label、value、unit、trend 颜色、deltaText
- AC-2.3 全部无数据时 chip 仍渲染但 `value="—"` `trend="none"`，不报错

### 4.3 US-3：手动刷新

**作为** 用户，**我需要** 在 AI 卡上点击 🔄 按钮，**以便于** 当我感觉数据已更新时立即看到新洞察（不等 30 分钟缓存）。

**AC**：
- AC-3.1 点击 🔄 后 ≤ 1.5s 看到新数据
- AC-3.2 刷新后 `generatedAt` 更新为当前时间
- AC-3.3 同一用户 1 分钟内连续点击 6 次后第 6 次返回 429，前端 Toast 提示

### 4.4 US-4：详情抽屉

**作为** 用户，**我需要** 点击 AI 卡或 ⓘ 按钮打开详情抽屉，**以便于** 在不离开首页的情况下看到完整 headline + chip 详情。

**AC**：
- AC-4.1 点击卡体或 ⓘ 按钮，el-drawer 打开，标题显示生成时间（如"AI 分析 · 2026-07-21 17:30"）
- AC-4.2 抽屉内显示完整 headline（与卡面一致）+ 3 个大尺寸 chip
- AC-4.3 抽屉可通过 X 按钮、ESC 键、点击遮罩三种方式关闭

### 4.5 US-5：空状态

**作为** 新用户（无任何数据），**我需要** 看到明确的引导文案，**以便于** 知道"录入更多数据后会看到洞察"。

**AC**：
- AC-5.1 无数据时 headline = "还没有数据，继续记录几天后将出现洞察。"
- AC-5.2 3 个 chip 均显示 `value="—"` `trend="none"`，无 deltaText
- AC-5.3 不出现 5xx 错误；HTTP 200

### 4.6 US-6：跨用户隔离

**作为** 用户 A，**我需要** 我的 AI 洞察不包含用户 B 的任何数据，**以便于** 保护个人隐私。

**AC**：
- AC-6.1 用户 A 的 insight headline/chip 数字仅来自 `t_task`/`t_plan`/... 中 `user_id=A` 的记录
- AC-6.2 切换到用户 B 登录后，AI 卡显示 B 的数据，与 A 完全不同
- AC-6.3 集成测试断言：用户 A 的 insight payload 不含任何 user B 的数据指纹

### 4.7 US-7：模块未上线降级

**作为** 项目维护者，**我需要** 在日报 v1.2.3 未合并到 main 时 AI 模块仍正常工作，**以便于** 独立 release 不被阻塞。

**AC**：
- AC-7.1 配置 `lp.ai.daily-enabled=false` 时，AI 卡正常返回 200，不报 503
- AC-7.2 DailyAiProvider 不向 `t_daily_report*` 发起任何查询
- AC-7.3 集成测试断言：daily-enabled=false 时，5 个 provider 中 4 个运行，1 个跳过

## 5. 功能范围与边界

### 5.1 In Scope（v2.0 必做）

| 模块 | 范围 |
|---|---|
| **后端 API** | `GET /api/v1/ai/insight/today`、`POST /api/v1/ai/insight/refresh`；沿用 `MyResponse<T>` 信封 |
| **数据源** | t_task（完成率）、t_plan（今日事件数）、t_expense（本周 vs 上周）、t_diet（今日摄入/目标）、t_daily_report*（连续生成天数，v2.0 默认 disabled）|
| **模板** | `ai-templates.properties` 集中 11+ 键值；JDK `String.format` 渲染 |
| **缓存** | Redis 30 分钟 TTL + 手动 refresh；不可用时降级为无缓存 |
| **错误码** | 沿用 1001/1003/1006；新增 1501（AI 整体不可用）|
| **限流** | today 60/min/用户，refresh 6/min/用户 |
| **前端** | AiInsightCard、AiChipItem、AiInsightDrawerView、aiInsight store、ai api 客户端；替换 HomeView 占位卡 |
| **测试** | 单元 38 + Slice 6 + 集成 5 + Vitest 12 + E2E 4 = 65 用例 |
| **文档** | 设计 spec（已完）+ 本 PRD（已完成）+ README 更新 + PR 描述 |

### 5.2 Out of Scope（v2.0 明确不做）

| 项 | 不做理由 |
|---|---|
| 独立分析页 `/ai-analysis` | v2.0 范围过大；先用抽屉收用户反馈 |
| 趋势图 / 图表库 | 与"轻量"原则冲突；图表库选型本身需独立 PRD |
| LLM 调用 | 显式不做；引入密钥/成本/限流新风险 |
| 多语言 i18n | 项目显式不做 |
| 推送 / 定时通知 | 与"打开首页触发"模式冲突 |
| 历史 insight 落表 | 落 `t_ai_insight` 需要独立 schema 设计 + 数据保留策略 |
| 用户自定义 chip 顺序 | 涉及前端状态管理 + 后端个性化配置；RICE 低 |
| 反馈机制 | 涉及反馈数据模型 + LLM 评估链路；v2.2+ |
| SMTP 邮件 | 引入第三方依赖；与"本地优先"原则冲突 |
| 跨设备同步 | 需 PWA / 推送 / 离线缓存；项目显式不做 |
| 团队/共享 | 项目显式不做 |

## 7. 风险与应对

### 7.1 技术风险

| # | 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|---|
| T1 | `DailyAiProvider` 接口无 `computeStreak` 方法 | 高 | T4.6 阻塞 | **实施前置**：在 daily 包加 `computeStreak(userId): MetricValue` 独立 commit；v2.0 启动时确认 |
| T2 | 模板表达力不足，未来扩展受限 | 中 | v2.1+ 阻碍 | 当前需求用 `String.format` + 简单条件足够；保留切换 Mustache 的可能性（v2.1 决策）|
| T3 | Redis 故障导致缓存长期失效 | 低 | 性能降级 | 不可用时降级到无缓存模式（§10.1 已设计）；无功能损失 |
| T4 | `expense` v1.2.1 / `daily` v1.2.3 未合 main 导致测试失败 | 中 | CI 红 | `isEnabled` 配置开关兜底；CI 启动前确认配置；本地开发可用 docker-compose 选配 |
| T5 | Testcontainers MySQL 启动慢（>30s）影响 CI | 中 | CI 总时长 +30s | 沿用项目已有 Testcontainers 配置；如慢可加 `reuse` 标志 |
| T6 | E2E Playwright 与 Element Plus 2.x el-drawer 兼容 | 低 | E2E 失败 | 沿用 [[ep-2x-playwright-el-select-helper]] 模式写 helper；现有 E2E 已有 drawer 经验 |
| T7 | `MetricValue` record 与 MyBatis-Plus 查询结果映射 | 低 | 编译/反序列化失败 | 保持 record 简单（4 字段）；不与 Mapper PO 直接映射 |
| T8 | 模板 properties 加载顺序 / 编码问题 | 低 | 中文乱码 | `application.yml` 强制 `UTF-8`；`ai-templates.properties` 文件头标注 |

### 7.2 资源风险

| # | 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|---|
| R1 | 1 人团队，时间被其他分支（v1.2.3）抢占 | 中 | v2.0 延期 | T1-T8 后端可与 v1.2.3 并行（不依赖）；T9-T11 需 v1.2.3 至少合入 daily 包接口 |
| R2 | 上下文丢失（连续上下文超 31k token） | 中 | 当次 session 需重开 | spec + PRD 已落档文档；恢复时直接读 spec.md + prd.md 即可继续 |
| R3 | CI runner 不稳定导致偶发失败 | 低 | 1-2 次重试 | 沿用项目已有 retry 机制 |

### 7.3 市场/产品风险

| # | 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|---|
| M1 | insight 准确度低于用户预期 | 中 | 用户觉得"AI 不准" | 纯规则 + 数据驱动，准确度可控；v2.1 收集反馈；v2.2+ 考虑 LLM 改写 |
| M2 | 用户不想看到 insight（觉得是噪音） | 低 | 关闭智能卡的需求 | 暂不做开关（YAGNI）；如 v2.1 反馈高频，加"折叠"按钮 |
| M3 | 与其他模块（日报）功能重叠 | 低 | 用户困惑"AI vs 日报"区分 | AI = 实时首页洞察；日报 = 周期性多维报告。文档/PR 描述明确边界 |
| M4 | 个人项目长期维护动力下降 | 中 | 半年后无人维护 | 文档完整 + 测试覆盖 → 易接手；不强求持续迭代 |

### 7.4 合规与安全风险

| # | 风险 | 可能性 | 影响 | 应对 |
|---|---|---|---|---|
| S1 | 日志泄露 PII | 低 | 隐私问题 | §10.2 已规定：仅打 userId；不打印 email/headline |
| S2 | 缓存被其他用户嗅探 | 极低 | 数据泄露 | 缓存键含 userId；Redis 配 ACL（项目既有）；TTL 30min |

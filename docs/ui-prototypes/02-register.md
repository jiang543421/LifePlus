## 1. 故事 → 视图要素映射

| AC | UI 必须呈现 |
|---|---|
| AUTH-1 AC1：注册成功后自动登录并跳到首页 | POST /auth/register 200 → 直接 issue access+refresh → 跳 `/` |
| AUTH-1 AC2：邮箱已注册时明确告知 | 错误码 1005 → 「该邮箱已注册」（注册是主动行为，反枚举交给登录兜底） |
| AUTH-1 AC3：密码强度不满足时列出未满足规则 | 错误码 1004 → 「密码需满足：长度 8-32 / 至少 1 个字母 / 至少 1 个数字」 |

### 1.1 与登录的反枚举边界

- **登录（1002）**：邮箱不存在/密码错 → 统一「邮箱或密码错误」（防枚举）
- **注册（1005）**：邮箱已注册 → 明确告知（注册是主动反馈，反枚举由登录兜底）
- 这是 PRD 显式选择的"非对称策略"，UI 层不做混淆

## 4. 字段清单（与后端 DTO 反查）

| 字段 | 必填 | 校验规则 | 后端 DTO |
|---|---|---|---|
| email | ✓ | 邮箱格式（前端基础 `@` 校验） | `RegisterRequest.email` (`@Email @NotBlank @Size(max=128)`) |
| password | ✓ | **前端实时校验** 3 条 + 后端 1004 兜底 | `RegisterRequest.password` (`@Size(8,32)` + 自定义 `@Pattern` 包含字母+数字) |
| nickname | ✗ | ≤ 32 字符 | `RegisterRequest.nickname` (`@Size(max=32)`) |

### 4.1 双层校验

| 层 | 何时校验 | 文案 |
|---|---|---|
| 前端 onInput | 用户输入过程 | 三条规则逐条 ✓/✗ |
| 前端 onSubmit | 点击注册 | 阻止提交直到全 ✓ |
| 后端 | POST /auth/register | 1004 兜底（防 devtools 绕过） |

**why 双层**：后端是 source of truth；前端反馈即时。

### 4.1.1 三条规则

1. 长度 8-32 位
2. 至少 1 个字母（大小写不限）
3. 至少 1 个数字

## 6. 错误码 → 文案映射

| code | 文案 | 触发 |
|---|---|---|
| 1004 | 密码需满足：长度 8-32 / 至少 1 个字母 / 至少 1 个数字 | 密码不合规 |
| 1005 | 该邮箱已注册 | 邮箱已注册 |
| 1006 | 请求过于频繁，请稍后再试 | 1 分钟内第 4 次 |
| 0 | （无提示，直接登录态） | 注册成功 |
| 其它 | 系统繁忙，请稍后再试 | 网络/5xx 兜底 |

## 7. 与 1.4 落地的关联

- `el-form` + `el-form-item`（password 用 `rules` prop + 自定义 validator）
- `authStore.register({email, password, nickname})` → `api/auth.ts → POST /auth/register`
- 成功后 `authStore.setSession(access, refresh, user)` → push `/`

## 8. 不在本原型范围

- 邮箱验证码（PRD Out of Scope）
- 邀请码 / 推广码（PRD 未提）
- 手机号 / 实名（PRD 显式不做，MVP1 仅邮箱+密码）
- 多语言切换（PRD Out of Scope）
- 第三方注册（PRD Out of Scope）
